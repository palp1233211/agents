## 功能说明

对需求文档进行结构化审查，生成待确认事项清单报告文件，供产品经理确认。

**输出位置**：`workspace/docs/{需求号}-{需求名}.md`

**命名规则**：文件名以需求号开头，便于检索和归档

### 责任范围

- **默认**：BI 后端负责的功能点 → 需要分析
- **排除**：明确标注 MS / HRS 负责的功能点 → 跳过分析

### 分析维度

| 维度 | 检测规则 | 示例 |
|------|----------|------|
| 硬编码数值 | 文档中出现具体数字、阈值、限制 | "超过 100 次限制" → 是否需要配置化？ |
| 模糊描述 | "优化、提升、改善、尽快、频繁" 等模糊词汇 | "提升用户体验" → 具体指什么？ |
| 边界缺失 | 描述了正常流程，但未说明异常/边界情况 | "显示订单列表" → 无数据时怎么展示？ |
| 排他规则 | 列举了 A、B、C，但未说明是否只有这些 | "支持微信、支付宝" → 不支持银行卡？ |
| 跨系统依赖 | 提到其他系统/模块名称 | "同步 ERP 库存" → 对接负责人是谁？ |
| 数据来源 | "显示xxx数据" | 数据从哪来？谁负责？ |
| 权限涉及 | 提到角色/权限相关 | 谁能看？谁能改？ |
| 前后矛盾 | 文档前后描述不一致 | 前面说"所有用户"，后面说"仅限VIP" |
| 引用缺失 | "见附录文档"但未提供链接 | 缺少具体引用位置 |
| 术语未定义 | 使用业务术语但未定义 | "活跃用户"的定义是什么？ |

## 输出格式

报告文件包含以下章节：

1. **责任范围识别**：区分 BI 后端负责的功能点
2. **待确认事项表格**：按十个维度列出问题
3. **重点澄清问题**：高/中优先级问题分类
4. **原始需求内容**：保留关键原文便于对照
5. **状态跟踪**：待确认/待评审/待开发状态

## 执行流程

1. **读取文档**：通过 lark-doc skill 获取飞书文档内容
2. **识别责任范围**：区分 BI 后端负责的功能点与其他团队功能点
3. **逐维度分析**：对 BI 后端负责的功能点进行十个维度分析
4. **生成报告**：写入 `workspace/docs/{需求名}.md` 文件
5. **输出摘要**：向用户展示报告路径和待确认事项数量
