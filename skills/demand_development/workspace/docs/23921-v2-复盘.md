## 根因分析（Root Cause）

### 问题清单

| 问题 | 根因 | 预防措施 |
|------|------|----------|
| 从 trunk 创建了错误分支 | 惯性思维（当前分支是 trunk），没有检查原需求分支是否存在 | 变更类需求开发前先 `git branch -a | grep 需求编号` 确认原分支 |
| TC-002 测试失败（aggregate 返回一行） | MySQL 无 GROUP BY 的聚合查询在无匹配时仍返回一行（COUNT=0） | 写聚合 SQL 时，无 WHERE 定位时结果集≠空，需手动判断 `total_count === 0` |
| TC-012 测试断言方向错误 | `sendReviewEmail` 内部 catch 了 Throwable，Mail 失败只 return false，不会向上抛 RuntimeException，导致"期望抛异常"的断言方向反了 | 写测试前先阅读被测方法的异常边界，区分"内部吞异常返回 false"和"向外抛异常"两种模式 |
| `customer_sub_group` 参数类型错误 | 最初传 `string`，但配置 JSON 的 key 是整型，PHP `isset($config[$strKey])` 在整型 key 上会失败 | 涉及 JSON/数组 key 查找时，明确 key 类型（int/string），保持一致 |

## 可复用模式（Reusable Patterns）

### 新增检查项

1. **变更类需求**：开工前先确认原需求分支，`git branch -a | grep 需求编号`，复用原分支，不创建新分支
2. **MySQL 聚合 SQL（无 GROUP BY）**：无匹配时仍返回一行，必须判断 COUNT 字段是否为 0 才能判断"无数据"
3. **测试异常方向**：写测试前先看被测方法的 catch 块——内部吞异常时用"返回 false 断言"，外抛时用"catch RuntimeException 断言"
4. **JSON config 的 key 类型**：PHP 数组 key 若来自 JSON（json_decode 整型 key 仍是整型），访问时参数类型必须匹配，`(int)` 强转
