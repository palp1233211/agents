### 改动文件

**`app/modules/Ph/BLL/AbnormalBLL.php`**

**改动位置**：`customerComplaintAssociativeInsertion` 方法，line 4978 之后

**改动内容**：

```php
// Before
$extra_info['complaints_type'] = $convert_after_sort;
$extra_info['complaints_sub_type'] = implode(',', $convert_after_arr);

// After
$extra_info['complaints_type'] = $convert_after_sort;

// 仅保留虚假揽件改约时间/取消揽件任务(2)和派件虚假留仓件/问题件(3)大类
if (!in_array($extra_info['complaints_type'], [2, 3])) {
    $logger->write_log('PH客户投诉：非虚假类大类，跳过入库。complaints_type：' . $extra_info['complaints_type'], 'info');
    return true;
}

$extra_info['complaints_sub_type'] = implode(',', $convert_after_arr);
```

**说明**：
- PH 专属模块文件，无需 `country_code` 守卫
- `return true` 与 `not_complaints_type()` 的跳过语义一致，避免 task 日志误报失败
- 过滤点在 `complaints_type` 解析完成后、事务开启前，覆盖所有后续入库路径

## 8. 方案自我校验

### 性能校验
- 查询复杂度：O(1)，仅 `in_array` 判断，无额外 DB 查询
- 无 N+1 风险，无新增索引需求

### 内存校验
- 无大数据加载，无风险

### 并发与一致性
- 入库幂等由原有 `$duplicate_removal` 机制保证，本次不变
- 无新增事务边界

### 资源与可观测性
- 新增日志：`PH客户投诉：非虚假类大类，跳过入库。complaints_type：{N}`
- 可通过日志统计过滤数量，监控上线效果

### 高风险清单

| 风险点 | 影响面 | 处理动作 |
|--------|--------|----------|
| `complaints_type` 解析为 null 时 `in_array` 行为 | 可能误放行 null | 实现时确认 null 不在 [2,3]，`in_array(null,[2,3])` 返回 false，已覆盖 |
| 其他国家误触发 | 多国数据异常 | PH 模块专属文件，物理隔离，零风险 |
